# Phase 4.6.2 — Graded Relevance Design

> Date: 2026-04-07 | Project: Ledger v2 | Phase: 4.6.2 | Branch: `feat/v2-phase-4.6.2-graded-relevance`

## Goal

Upgrade the Ledger golden dataset from binary judgments (`expected_doc_ids int[]`) to graded relevance using the TREC 4-level scale (0–3). Honest measurement: reward "good answers" instead of punishing everything that isn't the single canonical doc. Deliver a trustworthy ruler for every retrieval change that follows.

## Context

Run 12 (pre-cleanup) and run 13 (post-cleanup, new baseline) both show first-result accuracy stuck at ~65%. Per-query diagnosis of all 5 missed queries in run 12 showed:

- **3 of 5 misses are false negatives** — retrieval surfaced defensible alternative docs that the binary dataset doesn't count as correct
- **1 was a dataset bug** — an enumeration query that isn't a retrieval shape, deleted in run 13
- **1 is a genuine conceptual→jargon vocabulary gap** — real retrieval issue, deferred to a future reranker pass

Diagnosis is saved inline in the conversation transcript and will be summarized in the Ledger devlog when this phase completes. The conclusion: the 65% number is an under-measurement. Phase 4.6.2 gets us an honest number.

## Scale — TREC 4-level

| Grade | Name | Meaning |
|---|---|---|
| **0** | Not relevant | No useful info for this query. Wrong topic. |
| **1** | Related | Touches the topic but doesn't answer. |
| **2** | Relevant | Answers the query, but not the ideal/canonical source. |
| **3** | Highly relevant | The canonical, complete answer. |

**Why 4 levels, not 3 or 5:** NDCG was designed by Järvelin and Kekäläinen (2002) for multi-grade relevance with the formula `gain = 2^grade - 1` producing 0, 1, 3, 7 for grades 0–3. Our `eval-advanced.ts` already computes NDCG. 3 levels collapses "defensible alternative" and "canonical" into one grade, losing the distinction this phase exists to capture. 5+ levels hits judge fatigue without agreement gains.

**Decision heuristics for boundary calls:**
- **1 vs 2:** "Would a user be happy if this was the top result?" Yes = 2, No = 1.
- **2 vs 3:** "Is there a better doc for this query that I know exists?" Yes = 2, No = 3.

## Schema

### New table

```sql
CREATE TABLE eval_golden_judgments (
  id            bigserial PRIMARY KEY,
  golden_id     bigint    NOT NULL REFERENCES eval_golden_dataset(id) ON DELETE CASCADE,
  document_id   bigint    NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  grade         smallint  NOT NULL CHECK (grade BETWEEN 0 AND 3),
  judged_at     timestamptz NOT NULL DEFAULT now(),
  judged_by     text      NOT NULL DEFAULT 'adrian',
  notes         text,
  UNIQUE (golden_id, document_id)
);

CREATE INDEX idx_golden_judgments_golden_id   ON eval_golden_judgments(golden_id);
CREATE INDEX idx_golden_judgments_document_id ON eval_golden_judgments(document_id);
CREATE INDEX idx_golden_judgments_grade       ON eval_golden_judgments(grade);
```

| Column | Rationale |
|---|---|
| `id` | Stable PK for updates |
| `golden_id` | FK to query; CASCADE keeps judgments consistent with dataset edits |
| `document_id` | FK to documents; CASCADE removes grades against purged docs |
| `grade` | `smallint` + `CHECK` = cheap, constrained, validated at DB layer |
| `judged_at` | Audit — when was this judgment made |
| `judged_by` | Audit + forward-compat for multi-judge later |
| `notes` | Free-form reasoning for tricky cases |
| `UNIQUE (golden_id, document_id)` | One grade per query/doc pair |

### Changes to existing tables

**`eval_golden_dataset`:** drop `expected_doc_ids` column **after** the migration is complete and run 14 has landed successfully. Keep `query` and `tags`.

**`eval_runs`:** no DDL change. Store `hit_threshold` and `ndcg_gain_formula` inside the existing `config` jsonb via `CURRENT_SEARCH_CONFIG` in `eval-store.ts`.

### RPC functions

All transactional, aligned with the "business logic in SQL" principle:

- **`judgment_create(golden_id, document_id, grade, judged_by, notes)`** — insert. Errors if one exists.
- **`judgment_update(golden_id, document_id, grade, notes)`** — update existing grade, bump `judged_at`. Errors if row missing.
- **`judgment_delete(golden_id, document_id)`** — remove. Idempotent on missing row (no error).

No batch RPC. The rejudging tool writes one judgment at a time to keep every keystroke durable.

## Metric Definitions (Graded)

Single consistent rule across all rate metrics: **a result counts as good if its grade is ≥ 2**. Only NDCG uses the full grade gradation via `gain = 2^grade - 1`.

| Metric | Binary (old) | Graded (new) |
|---|---|---|
| Hit rate | Any returned doc in `expected_doc_ids` | Any returned doc with grade ≥ 2 |
| First-result accuracy | Top returned doc in `expected_doc_ids` | Top returned doc with grade ≥ 2 |
| Recall | `found / total_expected` | `(returned with grade ≥ 2) / (judgments with grade ≥ 2)` |
| MRR | `1 / position of first expected` | `1 / position of first grade ≥ 2` |
| NDCG | Binary gain (0 or 1) | `gain = 2^grade - 1` (0, 1, 3, 7) |
| Out-of-scope hit | Pass if zero results returned and `expected_doc_ids` is empty | Pass if zero results returned and no judgments at grade ≥ 2 |

### Expected metric shifts (hypotheses, not guarantees)

| Metric | Run 13 (binary) | Run 14 target | Why |
|---|---|---|---|
| Hit rate | 97.0% | ~98–99% | Already near ceiling |
| First-result accuracy | 65.2% | ~75–85% | Biggest win — defensible alternatives stop being false negatives |
| Recall | 81.4% | ~85–90% | Modest bump |
| MRR | 0.760 | ~0.80–0.85 | Follows first-result |
| NDCG | 0.769 | ~0.80–0.87 | Finally spreads meaningfully |

Hypotheses depend on how strict grading is. If run 14 goes *down* vs run 13 under these definitions, there is a scoring bug.

### Scoring rewrite — `scoreTestCase`

- Build `Map<document_id, grade>` from the test case's judgments
- Missing entries default to grade 0
- Hit = any returned doc has `grade >= 2`
- First-result hit = top returned doc has `grade >= 2`
- Recall = count of returned docs with `grade >= 2` divided by count of judgments with `grade >= 2`
- Position = index of first returned doc with `grade >= 2`
- MRR contribution = `1 / (position + 1)`
- NDCG gain per doc = `2^grade - 1`

### Out-of-scope semantics

A test case is out-of-scope when it has **zero judgments at grade ≥ 2**. All existing 12 out-of-scope cases (previously empty `expected_doc_ids`) remain out-of-scope post-conversion because the converter inserts no judgments for them.

### Metric report header

`formatReport` in `eval.ts` prints the threshold:

```
Search Evaluation — hit_threshold=2, ndcg_gain=2^g-1
```

### Unchanged code

- `eval-advanced.ts` (confidence intervals, calibration, coverage) — operates on `ITestResultProps`, which keeps its shape
- `eval-store.ts` save/load — same row shape, new config fields
- `compareRuns` / `formatComparison` — same metric names

## The Rejudging Tool — `ledger eval:judge`

### Commands

```
ledger eval:judge             # resume from lowest ungraded golden_id
ledger eval:judge --query 42  # jump to specific query
ledger eval:judge --progress  # print progress and exit
```

### Resume logic

Resume point = **lowest `golden_id` that has at least one top-10 search result without a judgment**. New queries added later are picked up naturally.

### Flow per query

1. Load test case with joined judgments
2. Run search via `searchHybrid` using current pipeline config
3. Print header with progress, query text, tags, already-graded docs with grades
4. Loop through top-10. For each doc without a judgment:
   - Print position, doc id, name, score, first ~200 chars of content
   - Prompt: `Grade [0/1/2/3]  s=skip  b=back  n=notes  ?=rubric  q=save & quit:`
5. On `0`/`1`/`2`/`3`: call `judgment_create` RPC immediately, advance
6. On `s`: skip (no DB write), advance
7. On `b`: rewind to previous doc in this query, show existing grade if any
8. On `n`: inline prompt for a note, attach to next judgment
9. On `?`: print the rubric inline with decision heuristics
10. On `q`: print session summary, exit cleanly
11. When all 10 graded: print "Query #N complete", advance to next query

### Durability

Every grade writes to the database immediately. No buffering. Cost: ~10ms per judgment. Benefit: zero work lost on crash or Ctrl+C. The right trade for human-input-speed tools.

### Progress display

On startup and after every query:

```
Progress: N / 132 queries complete (M%)
Judgments: X total (Y from binary conversion, Z new)
```

Two cheap SQL counts.

### File structure

```
src/commands/eval-judge.ts         ← new, thin command
src/lib/eval/eval-judge-session.ts ← new, session state + input handling
src/cli.ts                         ← register command
```

Per CLAUDE.md: commands stay thin, lib holds logic.

## Migration Sequence

Each step verified before the next. Rollback path stays open until step 9.

1. **Write migration SQL** (new table, indexes, RPC functions, pgTAP tests). Apply. Confirm pgTAP green.
2. **Write TypeScript types, loader, new scoring.** Unit tests green.
3. **Write `src/scripts/convert-judgments-to-graded.ts`**. Reads `expected_doc_ids`, inserts grade-3 judgments via the RPC. Idempotent.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Auto-conversion silently drops docs | Low | High | Step 4 manual count check. pgTAP tests on converter. |
| Scoring bug makes run 14 misleading | Medium | Medium | Step 5 dry-run parity check. Unit tests on `scoreTestCase`. |
| Judge fatigue → inconsistent grades | Medium | Medium | Rubric heuristics. Resumable tool for short sessions. |
| New queries added mid-rejudging break resume | Low | Low | Resume is "first ungraded," not "next index" |
| Dropping `expected_doc_ids` too early removes rollback | Low | High | Step 9 is LAST |
| Grading bias toward current pipeline's results | Medium (by design) | Low | Acknowledged. This phase fixes measurement, not recall. |

Top two watch-outs: **scoring bugs** (mitigated by step 5) and **judge fatigue** (mitigated by rubric heuristics and short resumable sessions).
