# Ledger v2: Production RAG System — Roadmap

> Date: 2026-03-29 | Status: In Progress | Project: Ledger

## What We're Building

Ledger started as a personal AI memory tool — notes stored in Supabase, searched by meaning via vector embeddings. v2 transforms it into a **production-grade RAG (Retrieval-Augmented Generation) system** that can:

- Store any type of content (text, PDFs, audio, images, code, web pages)
- Search using both meaning (vector) and exact words (keyword) combined
- Track every change with full rollback capability
- Support multiple users and agents with permissions
- Sync across machines in real-time
- Measure and improve search quality over time

## Why This Architecture

### The core insight: separate content from search

In v1, everything was in one table. A note's content, its metadata, and its search embedding were all in the same row. Chunked notes (split for better search) became multiple rows pretending to be one note.

In v2, we separate concerns:
- **`documents` table** — your actual content. One row per document, always complete.
- **`document_chunks` table** — search index. Small pieces of text with embeddings. Can be rebuilt anytime from the document content.

This is the same pattern used by LangChain, LlamaIndex, Pinecone, and every production RAG system. The document is the source of truth. The chunks are a search optimization.

### Why real columns instead of JSONB

v1 stored everything in a JSONB `metadata` blob. This caused:
- **Bad query plans** — Postgres can't track value distributions inside JSONB (hardcodes 0.1% selectivity estimate)
- **No constraints** — nothing prevented invalid domain values or missing required fields
- **Hidden data** — you had to know what was in the blob to use it
- **Slow filtered search** — every query parsed JSONB at runtime

v2 promotes every queryable field to a real column with proper types, indexes, and CHECK constraints. JSONB is gone entirely.

### Why 5 domains

Every document belongs to one of 5 domains that answer "what kind of thing is this?"

| Domain | What it stores | Syncs to every machine? |
|---|---|---|
| **system** | Ledger infrastructure (hooks, configs, sync rules) | Yes |
| **persona** | Your professional identity (personality, skills, conventions) | Yes |
| **workspace** | Your operational setup (dashboards, devices, dev environment) | Yes |
| **project** | Single codebase state (architecture, errors, status) | No — on demand |
| **general** | Personal knowledge (OAuth explainer, restaurants, book notes) | No — on demand |

Domains were designed through iterative discussion:
- Started with 4 (system, persona, workspace, project)
- Added `general` because personal knowledge doesn't fit any of the 4
- Workspace is purely informational — no extensions (skills, hooks, plugins) or identity docs (CLAUDE.md)
- Shared types (skill, hook, plugin-config) can exist in multiple domains with explicit defaults

Full domain spec: `docs/superpowers/specs/2026-03-28-v2-data-model-design.md`

### Why hybrid search

Vector search alone misses exact terms. Searching "pgvector HNSW" might not find a note containing those exact words if the vector embedding focuses on the broader concept of "database indexing."

Keyword search alone misses meaning. Searching "how does auth work" won't find a note titled "OAuth refresh token flow" because the words don't match.

Hybrid search runs both, then combines results using RRF (Reciprocal Rank Fusion) — documents found by both methods rank highest.

### Why transactional writes

Every write operation (create, update, delete) wraps the document change + chunk updates + audit log entry in a single Postgres transaction. If any part fails, everything rolls back. No orphaned documents without audit trails, no audit entries for failed writes.

### Why document-chunk separation for search

An embedding model produces better results on focused text (500-2000 chars) than on large mixed-topic documents (10,000+ chars). Chunking splits documents into smaller pieces, each with its own embedding. The search finds the most relevant chunk, then returns the full document for context.

Smart retrieval decides how much context to give the LLM:
- Small documents (under threshold) → return everything
- Large documents → return the matched chunk plus surrounding sections

## Database Architecture

**9 tables, 13 Postgres functions**

### Core tables

| Table | Purpose | Phase |
|---|---|---|
| `documents` | Source of truth — one row per document, all fields as columns | 1 |
| `document_chunks` | Search index — embeddings for vector/keyword search, FK to documents | 1 |
| `audit_log` | Change tracking — partitioned by year, survives document deletion | 1 |

### Infrastructure tables

| Table | Purpose | Phase |
|---|---|---|
| `agents` | Agent registry — permissions, API keys, JWT auth | 1 (schema), 6 (enforcement) |
| `embedding_models` | Model registry — tracks providers, dimensions, default model | 1 |
| `query_cache` | Embedding cache — reuse query embeddings for repeated searches | 1 (schema), 3 (wiring) |

### Feature tables

| Table | Purpose | Phase |
|---|---|---|
| `document_versions` | Full content snapshots at each version | 1 (schema), 7 (wiring) |
| `search_evaluations` | Search quality metrics — what was searched, what came back, was it useful | 1 (schema), 7 (wiring) |
| `ingestion_queue` | File processing pipeline — pending PDFs, audio, images | 1 (schema), 4 (wiring) |

### Postgres functions

| Function | What it does | Phase |
|---|---|---|
| `document_create` | Insert document + chunks + audit (transaction) | 1 |
| `document_update` | Update content + replace chunks + audit (transaction) | 1 |
| `document_update_fields` | Update document columns (typed params) + sync chunks domain + audit | 1 |
| `document_delete` | Soft delete + remove chunks + audit (transaction) | 1 |
| `document_purge` | Hard delete after grace period | 1 |
| `document_restore` | Undo soft delete | 1 |
| `match_documents` | Vector search via chunks → return documents | 1 |
| `match_documents_keyword` | Full-text keyword search | 1 (schema), 3 (wiring) |
| `match_documents_hybrid` | RRF fusion of vector + keyword | 1 (schema), 3 (wiring) |
| `retrieve_context` | Smart retrieval — full doc vs chunk + neighbors | 1 (schema), 3 (wiring) |
| `create_audit_partition_if_needed` | Auto-create next year's audit partition | 1 |
| `cleanup_query_cache` | Remove stale cached queries | 1 |
| `trg_documents_set_updated_at` | Auto-update timestamp trigger | 1 |

Full schema spec: `docs/superpowers/specs/2026-03-29-schema-rewrite.md`

## Phases

### Phase 1: Database Foundation
**Status:** Specced, ready to implement
**Spec:** `docs/superpowers/specs/2026-03-29-schema-rewrite.md`

Side-by-side migration: create new tables alongside old `notes` table, migrate data from old → new, verify everything works, then drop old tables. Ledger stays live throughout — no downtime, no context loss.

**Delivers:** The complete database that all future phases build on. Every table exists even if its TypeScript wiring comes later.

### Phase 2: Core TypeScript
**Status:** Specced
**Depends on:** Phase 1
**Spec:** `docs/superpowers/specs/2026-03-30-typescript-architecture.md`

Fresh TypeScript rewrite — 5 focused modules replacing the old 1000-line `notes.ts`:
- `document-classification.ts` — types, domains, protection, inference
- `document-operations.ts` — create, update, delete, restore (calls Postgres RPC)
- `document-fetching.ts` — getById, getByName, list, fetchSyncable (sync driven by is_auto_load, not domain)
- `ai-search.ts` — vector, keyword, hybrid search + smart retrieval
- `embeddings.ts` — embedding generation, chunking, caching, hashing
- New MCP tools (`*_documents`) alongside deprecated old ones (`*_notes`)
- All interfaces follow `INameProps` pattern with strong typing

### Phase 3: Search
**Status:** Not specced
**Depends on:** Phase 2
**Old Phase 2 spec (needs update):** `ledger-v2-phase-2-search` (#211)

Wire up the search functions that already exist in Postgres:
- Vector search via `match_documents` (already works from Phase 1)
- Keyword search via `match_documents_keyword`
- Hybrid search via `match_documents_hybrid` (RRF fusion)
- Smart retrieval via `retrieve_context` (full vs chunked)
- Query cache — check cache before calling OpenAI embedding API
- Query routing — decide vector vs keyword vs hybrid based on query analysis

Chunking improvements:
- Recursive chunking (headers → paragraphs → sentences)
- Semantic chunking (split by topic using embeddings)
- Chunk overlap for context continuity

Re-ranking:
- Cross-encoder re-ranker after initial retrieval (Cohere, Jina, or local model)

### Phase 4: Multi-Format Ingestion
**Status:** Not specced
**Depends on:** Phase 3

Build the ingestion pipeline for non-text content:
- PDF extraction (pdf-parse or Apache Tika)
- Image description (vision model — GPT-4o or Claude)
- Audio transcription (Whisper)
- Video processing (audio track + keyframe descriptions)
- Web page extraction (Firecrawl or cheerio)
- Code file parsing (with language-aware chunking)
- Spreadsheet/CSV processing
- Email extraction

Uses the `ingestion_queue` table for async processing. Each format gets an extractor that produces text → chunks → embeddings.

### Phase 5: Event-Driven Sync
**Status:** Planning
**Depends on:** Phase 2
**Old Phase 3 spec (needs update):** `ledger-v2-phase-3-sync` (#212)

Replace the deleted sync command with real-time event-driven sync:
- Supabase Realtime subscription (WebSocket)
- Daemon process (`ledger daemon`)
- LWW conflict resolution with audit trail
- File watcher for local changes
- Offline replay via audit_log

Realtime publication and REPLICA IDENTITY FULL are already set up in Phase 1.

### Phase 6: Access Control
**Status:** Planning
**Depends on:** Phase 5
**Old Phase 4 spec (needs update):** `ledger-v2-phase-4-access` (#213)

Per-agent authentication and permissions:
- JWT-based auth (replacing service role key)
- Agent registry (agents table already exists)
- RBAC permissions (from agents.permissions JSONB)
- RLS policies that read JWT claims
- Rate limiting via audit_log
- API key management CLI

### Phase 7: Observability & Polish
**Status:** Planning
**Depends on:** Phase 2
**Old Phase 5 spec (needs update):** `ledger-v2-phase-5-observability` (#214)

Measure and improve the system:
- Search evaluation tracking (search_evaluations table already exists)
- Document versioning (document_versions table already exists)
- Usage analytics (retrieval_count on documents, query_cache hit_count)
- Rollback from audit_log
- Automatic audit partition creation (function already exists)
- Cache management (cleanup function already exists)
- CLI stats commands
